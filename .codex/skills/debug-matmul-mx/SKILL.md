# MX/Cube Matmul Debugging

Use this skill to debug MX and cube matmul failures systematically. Treat it as an execution playbook, not as a hardware spec.

Do not start from `tilelang-st` cases that are already known-bad. First isolate whether the problem is:

- logical shape mismatch
- physical fractal/layout mismatch
- wrapper/lowering bug
- writeback bug
- simulator/runtime issue

## Quick Workflow

Follow this order unless there is a clear reason not to:

1. Synchronize shape facts across `kernel.pto`, `main.cpp`, and `golden.py`.
2. Run an all-ones sanity check to separate writeback bugs from layout bugs.
3. Isolate data paths and scale paths; do not debug all four inputs at once.
4. Read simulator middle-layer logs before inferring from final `C`.
5. If the case is generated by TileLang/PTODSL, compare its authoring IR and expanded IR against a passing handwritten micro-op case.
6. Only after the above, refine fractal mapping and scale pack details.

## Core Model

Keep these layers separate:

- Logical matrix shape: `A[M, K]`, `B[K, N]`, `C[M, N]`
- Physical source/destination layout: big-fractal order, small-fractal order, small-fractal size
- MTE transfer behavior: start position, step, stride, and transpose-like behavior of the transfer itself

Most “fractal looks wrong” conclusions are really caused by mixing these layers.

## First Checks

Before deep debugging, verify:

- `kernel.pto` uses the intended `M`, `N`, `K`
- `main.cpp` allocates matching input and output byte sizes
- `golden.py` uses matching matrix and scale shapes
- the runtime output file size matches the expected logical output size

If any of these disagree, simulator observations are not trustworthy.

For DSL-generated cases, add one more alignment layer:

- what the DSL author thought they called
- what the authoring IR actually contains
- what `vpto-expand-wrapper-ops` expands it into
- what the emitter finally packs into control operands

## Fast Sanity Check

Start with:

- `A` and `B` filled with one exactly representable value
- `A scale` and `B scale` filled with `1`

Use this to answer only these questions:

- Does `mad_mx` basically work?
- Does `mte_l0c_gm` write back the full result?
- Are `M/N/K` and output strides obviously wrong?

If all-ones already fails, do not spend time on intricate fractal reasoning yet.

## Common Failure Patterns

### `mte_l0c_gm` writeback bug

Suspect this first when:

- the first half of output is right and the rest is zero
- every block repeats or breaks at regular intervals

Do not assume `src_stride` or `dst_stride` should equal logical `M` or `N`.

### Data path correct, scale path wrong

Suspect this when:

- output is structured but numerically wrong
- rows or column groups repeat
- only certain `K-group` bands are wrong

Treat these independently:

- left data
- right data
- left scale
- right scale

Do not assume scale shares the same pack rule as data.

### “Passing pack” mistaken for spec

A `golden.py` implementation that passes only proves compatibility with:

- the current kernel
- the current shape
- the current simulator/runtime

It does not prove that the implementation is the canonical hardware interpretation.

If two implementations are byte-identical, prefer the one closer to the manual semantics.

### Source fractal size confused with grouped transfer behavior

For example, under `LOAD_L1_TO_L0B_2Dv2.b8`, a source small fractal may still be `16x32 / 512B`, while transpose-like transfer behavior groups multiple fractals during the move.

That grouped behavior changes:

- source address stepping
- destination landing pattern
- how to interpret source buffer packing

It does not necessarily mean the source fractal size itself changed.

## Isolate Data vs Scale

Use this debug sequence:

1. Pattern left data only; keep everything else uniform.
2. Pattern left scale only; keep everything else uniform.
3. Pattern right data only; keep everything else uniform.
4. Pattern right scale only; keep everything else uniform.

Avoid debugging all inputs with rich patterns simultaneously.

## Read Simulator Logs

Prioritize these logs:

- `core0.cubecore0.instr_log.dump`
- `core0.mte_log.dump`
- `core0.cubecore0.l1.rd_log.dump`
- `core0.cubecore0.l0a.wr_log.dump`
- `core0.cubecore0.l0b.wr_log.dump`

Use this workflow:

1. Find the target instruction in `instr_log`.
2. Record its `PC` and `ID`.
3. Match the same transfer in `mte_log` and read runtime parameters.
4. Correlate that transfer with `L1` reads and `L0A/L0B` writes.

Do not infer transfer semantics from the final `C` alone.

## Fractal Mapping Method

Use this method before reasoning from matmul outputs.

### Step 1. Label each source fractal

Fill each physical source fractal with a unique constant byte pattern.

This answers:

- which source fractals a transfer consumes
- where they land in the destination
- whether they land sequentially, interleaved, or after a physical transpose/reorder

This avoids contamination from scale or writeback bugs.

### Step 2. Separate block mapping from in-fractal mapping

Do it in two passes:

1. Fill each fractal with a unique constant.
2. Pick one fractal and fill it with row or column index patterns.

Use pass 1 to recover block-level mapping.
Use pass 2 to recover row-major, col-major, or transposed in-fractal mapping.

### Step 3. Trust runtime transfer parameters over guesswork

Extract these from `mte_log` when available:

- `src_addr`
- `dst_addr`
- `m_step`
- `k_step`
- `src_std`
- `dst_std`
- `transpose`
- `dtype`

If a layout theory contradicts the runtime transfer parameters, the theory is probably wrong.

### Step 4. Close the loop at the middle layers

Use this chain:

1. what bytes are in `v1.bin` / `v2.bin`
2. what L1 actually reads
3. what L0A / L0B actually contains
4. what `mad_mx` finally computes

This is more reliable than looking only at `C`.

### Step 5. Check equivalence by bytes

If two pack formulations look different, compare their raw bytes:

```python
np.array_equal(pack_a(...).reshape(-1), pack_b(...).reshape(-1))
```

If they are byte-identical, they are two expressions of the same physical layout.

## TileLang/PTODSL-Specific Workflow

Use this when the failing case is generated by TileLang or PTODSL rather than handwritten micro-op IR.

### 1. Compare against a passing handwritten case

Find a passing handwritten case with the same:

- shape
- dtype
- instruction family

Then compare:

- authoring op sequence
- expanded low-level ops
- runtime logs

Use this split:

- if both handwritten and DSL cases fail, suspect layout, `golden.py`, or instruction parameters
- if handwritten passes and DSL fails, suspect frontend, wrapper expansion, or emitter lowering

This was the key split for `tmatmul_mx`.

### 2. Inspect three IR layers

Always inspect:

- authoring/frontend IR
- IR after `vpto-expand-wrapper-ops`
- final emitter control packing

For MX cases, inspect these ops specifically:

- `pto.mte_l1_l0a_mx`
- `pto.mte_l1_l0b_mx`
- `pto.load_cbuf_to_ca_mx`
- `pto.load_cbuf_to_cb_mx`

Do not stop at surface operands like `(m, k)` or `(k, n)`. The real bug is often in derived controls:

- `x_start_position`
- `y_start_position`
- `x_step`
- `y_step`
- `src_stride`
- `dst_stride`

### 3. Prefer explicit control propagation

Wrapper expansion should derive explicit MX transfer controls.
Emitters should only pack those controls.

Do not let the emitter silently recreate high-level semantics from `m/k/n`.

If one side of the pipeline still recomputes stride/step/start ad hoc while the symmetric path already passes explicit controls through, treat that as a high-risk implementation smell.

### 4. Compare left and right MX lowering style

When `LowerCbufToCaMxOpPattern` and `LowerCbufToCbMxOpPattern` should be semantically symmetric but are implemented in very different styles, inspect the more special-case path first.

This often means:

- one side has clean control semantics
- the other still depends on historical heuristics

That was exactly the `A-scale` path issue in `tmatmul_mx`.

### 5. Do not emulate missing start-position semantics with stride hacks

If the real semantics need:

- source start position
- traversal step
- source and destination stride

then add start-position semantics to the surface or wrapper layer.

Do not hide them by overloading stride or step. That usually couples:

- full tiles
- tail tiles
- non-zero starts

into one brittle encoding.

### 6. For A5 `TMATMUL_MX`, treat logical `K=32` as physical `K=64` plus valid-shape semantics

This was the key fix for `fp8_e4m3_16x32x16`.

On A5, semantic MX matmul can legally express:

- lhs physical shape `[M, 64]` with valid shape `[M, 32]`
- rhs physical shape `[64, N]` with valid shape `[32, N]`
- `a_scale` physical shape `[M, 2]` with valid shape `[M, 1]`
- `b_scale` physical shape `[2, N]` with valid shape `[1, N]`

So if `main.cpp` says `k=32` but the kernel uses `mad_mx(..., K=64)`, do not assume that is a bug by itself.

Instead, ask whether the implementation is correctly representing:

- logical `K = 32`
- physical storage / transfer width `K = 64`

### 7. For A5 `TGEMV_MX`, separate GM source layout from final RHS tile layout

This was the key fix for `gemv_fp8_e4m3_e5m2_1x256x20`.

Do not conflate these three layers:

- GM logical source matrix `B[K, N]`
- source `mat` tile staged by `TLOAD`
- final `right` tile consumed by `TGEMV_MX` / `mad_mx`

The manual description like:

- RHS small fractal is `K0 x 16`
- small fractal inner layout is `ColMajor`
- small fractals are arranged `RowMajor`

describes the final `right` tile / L0B operand, not the GM source bytes.

Canonical `pto-isa` flow is:

1. GM logical `B[K, N]`
2. `TLOAD` to a source `mat` tile
3. `TEXTRACT` / `mte_l1_l0b` to the final `right` tile
4. `TGEMV_MX`

If you want to avoid `pto.textract` while debugging, use its equivalent VPTO sequence directly:

- `pto.mte_gm_l1_frac` for the `TLOAD`-equivalent GM -> source-mat stage
- `pto.mte_l1_l0b ... {transpose = true}` for the `TEXTRACT(Mat -> Right)`-equivalent stage

Do not treat a passing raw `load_cbuf_to_cb` sequence as the spec. It may only prove compatibility with one handwritten kernel.

### 8. For `TGEMV_MX` tail-`N`, `shape` is logical but `src_layout` is physical row stride

This was the main pitfall in the same gemv case above.

For `pto.mte_gm_l1_frac` on RHS source data:

- `shape(k, n)` should describe the logical valid rectangle
- `src_layout(...)` should describe the physical source row stride in GM

For example, when logical `N = 20` but the GM source rows are physically padded to `32`, the working setup was:

- `shape(256, 20)`
- `src_layout(32)`

and the GM input file remained a raw row-major `256 x 32` padded buffer.

Using `src_layout(20)` here produced a runnable kernel with wrong numerics, because the source-mat staging no longer matched what `mte_l1_l0b(k=256, n=20)` expects after its own logical-to-physical expansion.

### 9. For `TGEMV_MX` lhs, keep semantic `M=1` even if the load is physically 2D

Do not change `tgemv.mx` semantics to pretend lhs is `16 x K`.

The correct model is:

- semantic lhs valid shape is still `1 x K`
- physical `mte_l1_l0a` may still lower to a 2D CA load compatible with hardware block granularity

So it is valid to replace a handwritten raw `load_cbuf_to_ca` with semantic `pto.mte_l1_l0a` while keeping:

- lhs tile `valid_shape[0] == 1`
- `pto.tgemv.mx` operating on `M = 1`

### 7. In current VPTO ST flow, do not force `pto.tmatmul.mx` authoring if template expansion is unavailable

In this repo state, semantic verification and VPTO ST compilation are not the same question.

If `pto.tmatmul.mx` verifies but fails during wrapper expansion or kernel selection, do not keep pushing on the semantic op surface first.

Prefer this fallback for ST debugging:

- keep raw `mad_mx` with physical `K = 64`
- make GM-side inputs semantic and logical
- use `mte_gm_l1_frac nd2nz` to convert logical row-major padded GM data into the L1 mat layout that raw `mte_l1_l0a/l0b` already expects

This preserves semantic `logical K=32` without depending on a broken or unavailable `pto.tmatmul.mx` expansion path.

### 8. When `pto.tmatmul.mx` misses in ExpandTileOp, check DSL dtype spelling before deeper matcher surgery

The TileLang daemon query reports operand types in DSL-normalized names such as:

- `f8e4m3`
- `f8e5m2`

If the template registers MX kernels with ad-hoc spellings like:

- `ScalarType("f8E4M3FN")`
- `ScalarType("f8E5M2")`

then the expand path may still fail with:

```text
No kernel found for target='a5', op='pto.tmatmul.mx', operand_types=(f8e4m3, f16, ...)
```

Prefer registering MX template dtypes with the canonical DSL symbols:

- `pto.f8e4m3`
- `pto.f8e5m2`

before assuming the whole matcher stack is wrong.

### 9. Keep TileOp `tmatmul.mx` on logical `valid_shape K`; adapt hardware `K` in semantic-to-raw packing

`pto.tmatmul.mx` documentation and the C++ TileOp contract both use:

- `m = lhs.valid_shape[0]`
- `k = lhs.valid_shape[1]`
- `n = rhs.valid_shape[1]`

Do not “fix” the TileOp template by changing semantic `k` to `lhs.shape[1]`.
That mixes tile-level semantics with hardware packing.

For the `fp8_e4m3_16x32x16` case:

- logical `K = 32`
- physical storage/load width `K = 64`
- current A5 `MMAD.MX` still wants `X_t.K = 64`

So if wrapper expansion fails with:

```text
The K of MMAD_MX instr must be a multiple of 64, K=32
```

the correct repair point is:

- keep `pto.tmatmul.mx -> pto.mad_mx(..., k=valid_shape_k)` in the template
- normalize MX `k` to the current hardware-consumed physical multiple when packing raw `X_t`

This keeps TileOp semantics aligned with docs and moves target-specific width adaptation into the semantic-to-raw layer.

### 10. When a broad ST testcase is blocked by unrelated legacy cases, fork a minimal semantic testcase first

If the original ST directory cannot build because unrelated cases fail in lowering
for reasons such as legacy `pto.addptr` GM pointer arithmetic, do not keep
debugging the new semantic case inside that noisy file first.

Prefer creating a dedicated minimal testcase that contains only:

- one target case
- the exact raw DMA path you want to preserve
- the semantic TileOp you want to validate

This gives a clean answer to:

- does the expand path compile?
- does the runtime execute?
- is the remaining failure in scale payload semantics only?

## FP4 Packed-Byte MTE Checks

Use this checklist when `tmatmul_mx` or TileOp FP4 cases fail precision while
similar FP8/BF16 cases pass.

### 1. Treat FP4 source shapes as packed-byte matrices for low-level MTE

FP4 x2 data has two logical FP4 values per byte. For low-level
`pto.mte_gm_l1_frac`, express `shape` and `src_layout` in the byte matrix that
is physically present in GM, not in logical FP4 element counts.

Examples from A5 `tmatmul_mx`:

- logical/padded `A[M, K_aligned]` FP4 becomes byte matrix
  `A_bytes[M, K_aligned / 2]`
- logical/padded `B[K_aligned, N_padded]` FP4 becomes byte matrix
  `B_bytes[K_aligned, N_padded / 2]`

So a pto-isa `TLOAD` over logical `A[128, 118]` with padded `K=128` maps to a
PTOAS low-level load like:

- `shape(128, 64)`
- `src_layout(64)`

and logical `B[118, 64]` with padded `K=128, N=64` maps to:

- `shape(128, 32)`
- `src_layout(32)`

Do not feed logical FP4 dimensions such as `118` or `64` directly into a raw
byte-level transfer without checking whether the op expects bytes or elements.

### 2. Prefer `mte_gm_l1_frac ... nd2nz` for row-major padded FP4 data

If GM data is generated as padded row-major packed FP4 bytes, a raw
`mte_gm_l1` only copies bytes; it does not create the L1 NZ/fractal layout that
later `mte_l1_l0a` / `mte_l1_l0b` expect.

The working pattern is:

- stage FP4 data with `pto.mte_gm_l1_frac ... nd2nz`
- then use the existing raw `mte_l1_l0a` / `mte_l1_l0b` path
- keep scale loads raw when `gen_data.py` already emits scale payloads in the
  packed layout consumed by the MX scale load path

This mirrors the pto-isa `TLOAD`/ND2NZ flow more closely than a byte-for-byte
GM-to-L1 copy.

### 3. Size B data and L1 scale offsets from padded physical bytes

For FP4 RHS data, compute the staged byte count from padded physical storage:

```text
B_bytes = K_aligned * N_padded / 2
```

Do not use only the logical valid rectangle. In failing A5 cases, host code
allocated/copied a full padded FP4 RHS buffer such as `2048B`, while the `.pto`
kernel copied only `1024B`. The next L1 region, often B scale, then started too
early and overlapped or truncated B data.

When checking this class of bug:

- compare `main.cpp` allocation/copy sizes against `.pto` GM->L1 copy sizes
- move B scale L1 offset after the full padded B data region
- split GM->L1 copies into multiple transfers when a single transfer size limit
  requires it, for example `2 x 1024B`
- scale copies may also need looped copies, for example `2 x 64B`, depending on
  the padded scale payload

This was the precision root cause for small/tail FP4 `tmatmul_mx` cases such
as `fp4_e2m1_e1m2_115x64x30` and `fp4_e2m1_4x30x8`.

### 4. Use pto-isa/Bisheng comparison to localize the layer

When a pto-isa FP4 case passes, dump or inspect its lowering and compare:

- whether data enters L1 through `TLOAD`/ND2NZ-like staging
- the logical `TLOAD` shape versus the packed-byte low-level shape
- L1->L0A/L0B S4 transfer counts and offsets
- `MMAD.MX` immediate/control operands

If L1->L0B S4 transfers and `MMAD.MX` controls already match the passing
pto-isa case, suspect the GM->L1 data contract or host/L1 byte sizing before
changing cube op parameters.

## MX Scale Lowering Checks

Use this checklist when the result looks like scale was read incorrectly.

### 1. Do not equate logical shape with source stride

Manual text such as `Scale Matrix A is M x K/32` only describes logical element count.

It does not directly define:

- physical L1 source stride
- `load_cbuf_to_ca_mx` / `load_cbuf_to_cb_mx` source stride
- L0A/L0B consumption step

To derive real transfer stride, first fix:

- small-fractal byte size
- big-fractal order
- small-fractal order
- which dimension the transfer walks

### 2. Do not assume data-path success implies scale-path success

These can be independently correct or wrong:

- `mte_l1_l0a` / `mte_l1_l0b`
- `mte_l1_l0a_mx` / `mte_l1_l0b_mx`
- `mad_mx`

Data-path success does not prove MX scale lowering correctness.

### 3. Validate continuous consumption region first

Before deriving stride from logical scale shape, ask:

- what address range does cube actually consume continuously from L0A/L0B?

If lowering writes scale with large address jumps but cube only consumes a dense low-address region, expect:

- data to look fine
- logical scale shape to look plausible
- results to behave as if scale were repeated or misread

### 4. Keep start-position semantics explicit

For stable and testable MX scale loads, `start_row` and `start_col` should be first-class semantics.

Do not try to reconstruct them later inside the emitter.

### 5. Do not expose `transpose` on MX load surfaces

For current semantics:

- `mte_l1_l0a` / `mte_l1_l0b` may accept `transpose`
- `mte_l1_l0a_mx` / `mte_l1_l0b_mx` should not expose `transpose`

If an MX failure tempts you to add a transpose flag, stop and re-check:

- source layout
- explicit control derivation
- simulator middle-layer logs

## Validated Reference Case: `mad_mx_32x128x64`

Treat this section as a reference point, not as a universal rule.

### Logical shape

- `M = 32`
- `N = 64`
- `K = 128`

### Expected file sizes

- `v1.bin`: `4096` bytes
- `v2.bin`: `8192` bytes
- `v4.bin`: `128` bytes
- `v5.bin`: `256` bytes
- `v3.bin` / `golden_v3.bin`: `32 * 64` `f32`

### Validated left pack

Left data:

```python
def pack_lhs_mx_fp8_fractal(matrix):
    return matrix.reshape(m // 32, 32, k // 32, 32).transpose(2, 0, 1, 3)
```

Left scale:

```python
def pack_a_scale(a_scale):
    return a_scale.reshape(
        m // 16, 16, k_groups // 2, 2
    ).transpose(0, 2, 1, 3).reshape(-1)
```

### Validated right pack

Right data:

```python
def pack_rhs_mx_fp8_fractal(matrix):
    return matrix.reshape(k // 16, 16, n // 32, 32).transpose(2, 0, 1, 3)
```

Right scale:

```python
def pack_b_scale(b_scale):
    return b_scale.reshape(
        k_groups // 2, 2, n // 16, 16
    ).transpose(2, 0, 3, 1).reshape(-1)
```

This corresponds to:

- `B` data: big fractal `col-major`, each `512B` source fractal internally `16x32 row-major`
- `B` scale: big fractal `col-major`, small fractal `col-major`

### Case-specific symptoms that were validated

- If `mte_l0c_gm.src_stride` is incorrectly written as `N = 64`, the first `32` columns may be correct and the last `32` columns may become zero.
- If left `A-scale` group order is wrong, rows `0..15` and `16..31` can appear duplicated or aliased.
- For right `B` layout debugging, the fastest method was labeling each `512B` source fractal and checking the corresponding `l0b.wr_log`.

## Validated Small-Case Recipe: `fp8_e4m3_16x32x16`

Use this recipe when a small MX matmul case has semantic `K=32` but physical `K=64`.

### Working implementation pattern

- Keep kernel-side compute as raw `mad_mx(..., M=16, N=16, K=64)`.
- Load `A` and `B` from GM with `mte_gm_l1_frac nd2nz`, not raw `mte_gm_l1`.
- Keep scale loads on the existing raw MX path unless evidence shows scale is also wrong.

### Working GM contract

For this specific case, GM data should be:

- `A`: row-major padded `16x64`
- `B`: row-major padded `64x16`

Do not pre-pack `A` and `B` into MX fractal order in `gen_data.py` for this case.

The raw micro-op sequence already converts:

- GM row-major padded data
- into L1 mat layout via `mte_gm_l1_frac nd2nz`
- then into L0A/L0B via `mte_l1_l0a` / `mte_l1_l0b`

### Fast failure signal

If all-ones or simple patterned input produces results such as:

- every output element becoming `16`
- alternating correct and zero rows

suspect the GM-to-L1 data contract first, before blaming:

- MX scale packing
- `mad_mx`
- writeback

### Reference preference

For MX raw behavior, prefer:

- `test/vpto/cases/micro-op/cube-matmul/mad_mx_32x128x64`

Do not treat `tgemv_mx` as a reference for this path unless it has been independently validated.
